# 电子钱包补充数据接入方案

更新时间：2026-03-15

## 1. 背景

现有系统的主分析链路以银行流水为核心，入口是 `api_server.py`，真理源是 `output/cleaned_data/`。  
在实际核查中，微信/支付宝等电子钱包数据通常晚于银行流水取得，因此它们不能强行塞进现有银行清洗链，而应作为“后补协查数据”接入。

本方案基于 2026-03-15 对桌面 `ZFB+WX` 样本包的实物盘点形成。该样本包当前包含：

- `1` 个协查回函 PDF
- `1` 个支付宝注册信息 Excel
- `1` 个支付宝账户明细 Excel
- `14` 个微信注册信息 TXT
- `14` 个微信登录轨迹 TXT
- `22` 个财付通注册信息 TXT
- `22` 个财付通交易明细 TXT

按当前解析器直接扫描样本目录，可得到一组稳定的样本级统计：

- `10` 个电子钱包主体摘要
- `14` 个支付宝实名账户
- `25,154` 笔支付宝交易
- `12` 个微信账号
- `22` 个财付通账号
- `4,600` 笔财付通交易
- `7,766` 条微信登录轨迹

这批数据对系统而言是“盲盒”输入：我们知道它是围绕当前排查对象提供的补充数据，但事先不知道目录结构、字段映射和主体对应关系。因此第一原则不是“直接分析”，而是先做结构化、归档和映射。

## 2. 样本数据结构

### 2.1 支付宝数据

支付宝样本是一套 Excel 主表，特征稳定，适合作为结构化补充流水源：

- `*_注册信息_*.xlsx`
  - 每行一个支付宝实名账户
  - 关键字段包括：`用户ID`、`账户名称`、`证件号`、`绑定手机`、`注册时间`、`绑定银行卡`
- `*_账户明细*.xlsx`
  - 以身份证号为 sheet
  - 每行一笔交易
  - 关键字段包括：`交易创建时间`、`交易对方信息`、`金额（元）`、`收/支`、`交易状态`、`交易来源地`、`类型`

结论：

- 支付宝数据天然带有“实名主体 + 交易明细”的双层结构
- 它不是银行流水标准列，但足够稳定，可以抽象成“电子钱包交易补充源”
- 最重要的锚点是：`证件号`、`绑定手机`、`账户名称`

### 2.2 微信 / 财付通数据

微信样本不是单一表格，而是按“手机号 / 身份证号 / 微信号别名”展开的证据目录：

- `WX*/IPHONE/<手机号>/regInfobasicInfo（公开）.txt`
  - 微信注册基础信息
  - 常见字段：`微信号`、`别名`、`昵称`、`绑定手机号`、`注册时间`
- `WX*/IPHONE/<手机号>/WX登录轨迹（公开）.txt`
  - 微信登录日志
  - 常见字段：`时间`、`IP`
- `TP*/IDCARD/<身份证号>/<账号标识>/TenpayRegInfo（公开）.txt`
  - 财付通注册信息
  - 常见字段：`账号`、`注册姓名`、`注册身份证号`、`绑定手机`、`开户行信息`、`银行账号`
- `TP*/IDCARD/<身份证号>/<账号标识>/TenpayTrades（公开）.txt`
  - 财付通交易明细
  - 常见字段：`借贷类型`、`交易业务类型`、`交易用途类型`、`交易时间`、`交易金额(分)`、`对手侧账户名称`

结论：

- 微信数据是“多目录证据包”，而不是单一流水表
- 主体匹配不能只靠文件名，必须联合使用 `身份证号`、`手机号`、`微信别名`、`wxid`、`注册姓名`
- 财付通交易明细是最接近可分析流水的数据层
- 微信注册信息和登录轨迹更偏“身份补强”和“行为轨迹补强”

### 2.3 当前样本的真实目录骨架

`ZFB+WX` 当前不是扁平目录，而是两套来源并存：

```text
ZFB+WX/
├── 调证协查回函...pdf
├── 768219766_1770426397138-8-masd100%/
│   ├── 768219766_注册信息_*.xlsx
│   └── 768219766_账户明细*.xlsx
└── L-1770258514959-8-masd100%/
    ├── WX*/IPHONE/<手机号>/regInfobasicInfo（公开）.txt
    ├── WX*/IPHONE/<手机号>/WX登录轨迹（公开）.txt
    ├── TP*/IDCARD/<身份证号>/<wxid或别名>/TenpayRegInfo（公开）.txt
    └── TP*/IDCARD/<身份证号>/<wxid或别名>/TenpayTrades（公开）.txt
```

可以据此确认几个关键规律：

- 支付宝是“单批 Excel 包”，目录名更像批次号或协查任务号
- 微信目录按手机号展开，财付通目录按身份证号再展开到账号别名或 `wxid`
- `WX*` 与 `TP*` 目录本身不携带实名，实名锚点主要在文件内容里
- 微信和财付通之间真正的桥是：`手机号`、`别名`、`wxid`、`绑定银行卡`

## 3. 业务定位

电子钱包数据在业务上不应该替代银行流水，而应该承担以下补充职责：

1. 补第三方支付盲区  
   现有银行流水里只看到“支付宝/微信/财付通”通道金额，看不到钱包内部对手方和用途；电子钱包数据用于把通道拆开。

2. 补实名映射链  
   通过 `证件号`、`手机号`、`微信号别名`、`绑定银行卡` 建立“银行账户 <-> 支付宝 <-> 微信 <-> 财付通”的映射。

3. 补交易对手与消费场景  
   识别红包、转账、扫码付款、商户消费、还款、零钱/余额支付等细分行为。

4. 补行为轨迹  
   微信登录时间和 IP 可作为异地登录、时间异常、账号控制关系的补充证据。

5. 补核查建议闭环  
   现有报告只会提示“第三方支付占比高，建议调取电子钱包明细”；接入后应把“建议调取”升级为“已取得并已结构化分析”。

## 4. 推荐目录策略

### 4.1 推荐放置位置

推荐把后补电子钱包数据放在输入目录下的独立子目录：

```text
<inputDirectory>/
  补充数据/
    电子钱包/
      批次_20260315_ZFB_WX/
        ...
```

### 4.2 为什么不能直接散放在 `data/` 根目录

- 现有系统会递归扫描 PDF 和 Excel
- 协查回函 PDF 可能被误识别为普通线索文件
- 支付宝 Excel 虽然不会进入银行主清洗链，但会增加无意义的扫描噪声

因此需要两层保护：

- 目录规范：电子钱包数据单独放在 `补充数据/电子钱包/`
- 程序豁免：主链扫描器跳过电子钱包补充目录

## 5. 接入原则

### 5.1 不进入 `cleaned_data/`

电子钱包数据不应写入 `output/cleaned_data/`，原因是：

- 字段体系与银行流水不同
- 取得时点通常滞后
- 与主银行流水混写会破坏“银行清洗链”的稳定性

### 5.2 进入 `analysis_cache/` 的补充缓存

当前已经落地的接入链路如下：

1. `wallet_data_extractor.py` 扫描补充目录并输出 `walletData.json`
2. `api_server.py` 在主分析 Phase 4 中提取电子钱包数据，并在主体映射完成后执行二次归并
3. 电子钱包摘要会挂到 `profiles[*].wallet_summary`
4. 第一批电子钱包预警写入 `suspicions.wallet_alerts -> walletAlerts`
5. 聚合器把 `wallet_summary` / `wallet_alerts` 纳入实体证据包和综合风险评分
6. 图谱缓存会补充电子钱包对手方边，并在图谱页展示“电子钱包预警 / 对手方”区块
7. 正式报告会在个人章节生成“电子钱包补充核查”段落
8. 前端同时提供：
   - “补充数据”标签页，查看主体摘要
   - “风险情报”中的“电子钱包预警”筛选

因此，电子钱包数据目前已经不只是“缓存可见”，而是形成了完整的补充层闭环：

- `walletData.json` 作为结构化缓存
- `wallet_summary` 作为个人画像补强
- `walletAlerts` 作为风险侧入口
- 聚合证据包 / 综合风险评分作为排序入口
- 图谱中的电子钱包关系边作为可视化入口
- HTML / TXT 报告作为结果落地

### 5.3 晚到数据的处理原则

电子钱包数据属于“阶段性补录数据”，应满足：

- 没有电子钱包目录时，主程序行为保持不变
- 有电子钱包目录时，只增强补充结果，不推翻已有银行分析
- 前端将其展示为“补充层”而不是“主链重算结果”

## 6. 当前使用逻辑

当前版本对电子钱包数据的使用逻辑分成五层，且全部遵守“补充层增强、不回写主链”的原则。

### 6.1 结构发现层

先识别文件类型，再决定解析器，不依赖用户手工配置：

- 支付宝注册表 -> 实名账户层
- 支付宝账户明细 -> 交易层
- 微信注册信息 -> 账号映射层
- 微信登录轨迹 -> 行为轨迹层
- 财付通注册信息 -> 账号绑定层
- 财付通交易明细 -> 交易层

### 6.2 主体归并层

归并锚点按优先级组合使用：

1. `身份证号`
2. `姓名 / 注册姓名`
3. `手机号`
4. `微信别名 / wxid / 财付通账号`
5. `绑定银行卡`

这使系统即便面对“盲盒目录”，也能先把碎片证据拼成“主体摘要”。

### 6.3 补充画像层

对每个主体，目前沉淀以下补充信息：

- 支付宝账户数、交易笔数、收支总额、主要对手方
- 微信账号数、财付通账号数、财付通交易笔数
- 微信登录轨迹数量、最近登录时间
- 跨平台手机号 / 银行卡 / 别名重叠信号

这些内容进入 `walletData.json`，并按人挂到 `profiles[*].wallet_summary`。

### 6.4 补充预警层

当前已落地的第一批电子钱包规则包括：

- 电子钱包总体交易规模过大
- 跨平台账号绑定关系明确
- 高频或大额电子钱包对手方
- 无法自动归并的微信账号

这些规则不会改写原有银行疑点，而是追加到 `walletAlerts` 供前端和报告消费。

### 6.5 展示与报告层

当前落地的展示位置：

- 前端“补充数据”页：看主体摘要、样本规模、未归并微信账号
- 前端“风险情报”页：看电子钱包预警
- 前端“关系图谱”页：看电子钱包对手方边、电子钱包预警和已入图的补充关系
- HTML / TXT 报告：看“数据来源及完整性声明”中的第三方支付状态，以及个人章节中的“电子钱包补充核查”
- 线索聚合：看电子钱包证据是否推高实体综合风险分

## 7. 后续迭代方向

第二阶段已完成的事项：

- 将电子钱包摘要纳入正式报告
- 为“第三方支付是否已补数”提供状态化文案
- 在风险情报页增加电子钱包预警筛选
- 将电子钱包摘要和预警纳入聚合证据包与综合风险评分
- 将电子钱包对手方边纳入图谱缓存和图谱页展示

下一阶段建议继续推进：

1. 扩展电子钱包规则库  
   增加扫码转账、红包密集往来、商户与自然人混收、深夜高频支付等规则。

2. 增强证据钻取  
   从报告或前端直接回看命中预警的原始 TXT / Excel 行。

3. 引入时间异常联动  
   将登录轨迹和财付通交易时间一起纳入时序异常检测。

## 8. 当前实现边界

当前已经完成：

- 文档归档与样本结构说明
- 主链扫描器对电子钱包补充目录的豁免
- 电子钱包数据解析与 `walletData.json` 缓存
- 画像挂接 `wallet_summary`
- 风险页 `walletAlerts`
- 聚合证据包与综合风险评分接入
- 图谱缓存与图谱页接入
- HTML / TXT 报告中的电子钱包补充核查段落

当前仍未完成：

- 电子钱包交易并入银行流水统一建模
- 图谱中的路径解释与电子钱包证据回溯增强
- 订单号、聊天记录、商户凭证等更深层交叉取证
