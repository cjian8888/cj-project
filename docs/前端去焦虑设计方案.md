## 问题背景

### 当前状况

| 指标 | 数值 |
|------|------|
| `profiles.json` 大小 | **43.1 MB** |
| 内网传输时间 (100Mbps) | ~3.4 秒 |
| 本地回环传输 (1Gbps) | ~0.3 秒 |
| JSON 解析时间 | 1-8 秒 |

### 用户体验问题

1. **无进度反馈**: 页面初始化时调用 `api.getResults()` 加载 43MB 数据，用户只看到静态提示"正在检查后端缓存数据..."
2. **等待焦虑**: 3-5 秒的静默加载时间，用户不知道程序是否在正常工作
3. **缺少预期**: 没有告知用户数据量和预计加载时间

---

## 设计方案

### 方案一：分块加载 + 真实进度反馈 ⭐推荐

**核心思想**: 后端流式传输，前端实时显示真实进度

#### 后端改造

```python
# api_server.py 新增端点
@app.get("/api/results/stream")
async def get_results_stream():
    """流式返回分析结果，支持进度跟踪"""
    cache_dir = os.path.join("output", "analysis_cache")
    file_path = os.path.join(cache_dir, "profiles.json")
    total_size = os.path.getsize(file_path)
    
    chunk_size = 1024 * 1024  # 1MB 每块
    
    def generate():
        with open(file_path, 'rb') as f:
            while True:
                chunk = f.read(chunk_size)
                if not chunk:
                    break
                progress = f.tell() / total_size
                yield f"data: {json.dumps({'progress': progress, 'phase': '数据传输中...'})}\n\n"
    
    return StreamingResponse(generate(), media_type="text/event-stream")
```

### 方案二：模拟进度 + 骨架屏 🚀快速实施

**核心思想**: 保持现有 API，添加模拟进度动画和更友好的 Loading UI

#### 改造点

1. **AppContext.tsx** - 添加渐进式进度更新

```typescript
const loadCachedData = async () => {
    let progressInterval: NodeJS.Timeout;
    
    try {
        setAnalysis(prev => ({ 
            ...prev, 
            isLoading: true,
            progress: 0,
            currentPhase: '正在连接后端服务...' 
        }));
        
        // 阶段 1: 连接检查 (0-10%)
        const status = await api.getStatus();
        setAnalysis(prev => ({ ...prev, progress: 10 }));
        
        if (status.status === 'completed') {
            // 阶段 2: 开始传输 (10-20%)
            setAnalysis(prev => ({ 
                ...prev, 
                currentPhase: '正在加载分析数据...',
                progress: 20 
            }));
            
            // 模拟进度更新 (20-90%) - 非线性增长
            progressInterval = setInterval(() => {
                setAnalysis(prev => {
                    if (prev.progress >= 90) return prev;
                    // 越往后增长越慢，模拟真实加载曲线
                    const increment = Math.max(0.5, (90 - prev.progress) / 15);
                    return {
                        ...prev,
                        progress: Math.min(prev.progress + increment, 90),
                        currentPhase: `正在加载分析数据... ${Math.floor(prev.progress)}%`
                    };
                });
            }, 300);
            
            const result = await api.getResults();
            clearInterval(progressInterval);
            
            // 阶段 4: 数据处理 (90-100%)
            setAnalysis(prev => ({ 
                ...prev, 
                currentPhase: '正在处理数据...',
                progress: 95 
            }));
            
            // ... 数据合并逻辑
            
            setAnalysis(prev => ({ ...prev, progress: 100 }));
        }
    } catch (error) {
        clearInterval(progressInterval);
        // 错误处理...
    }
};
```

3. **修改 EmptyState.tsx 或新增页面级 Loading**

在数据加载期间显示全屏 Loading 覆盖层，而不是仅显示在侧边栏。

**优点**: 
- 实现成本低（约 2 小时）
- 无需后端改造
- 用户体验显著提升

**缺点**: 
- 进度是模拟的，不完全准确
- 网络波动时体验可能不一致

### 方案三：延迟加载 + 优先级策略 📦长期规划

**核心思想**: 先加载核心数据，再异步加载详细数据

#### 数据分层

```typescript
interface DataState {
    // Tier 1: 核心数据（先加载，<100KB）
    persons: string[];
    companies: string[];
    summary: {
        totalIncome: number;
        totalExpense: number;
        transactionCount: number;
    };
    
    // Tier 2: 详细数据（后加载，43MB）
    profiles: Record<string, Profile>;
    suspicions: SuspicionResult;
    analysisResults: AppAnalysisResults;
}
```

#### 后端改造

拆分 API 端点：
- `GET /api/results/summary` - 轻量级摘要
- `GET /api/results/details` - 详细数据

#### 前端改造

```typescript
const loadData = async () => {
    // 第一步：加载摘要（快速）
    const summary = await api.getSummary();
    setData(prev => ({ ...prev, ...summary }));
    
    // 页面可交互，显示骨架屏
    
    // 第二步：后台加载详细数据
    const details = await api.getDetails();
    setData(prev => ({ ...prev, ...details }));
};
```

**优点**: 
- 首屏加载极快
- 渐进式体验最佳

**缺点**: 
- 需要大量前后端改造
- 组件需要支持"无数据"状态

## 注意事项

1. **超时设置**: `profiles.json` 增大后，需要确保 API 超时时间足够（建议 30-60 秒）
2. **错误处理**: 加载失败时给出明确的重试指引
3. **兼容性**: 确保改造不影响现有功能
4. **性能**: 模拟进度间隔不宜过短，避免不必要的渲染
