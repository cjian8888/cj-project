## 二、问题1：程序的默认归集是如何取得的？

### 2.1 默认归集获取入口

```python
# PrimaryTargetsService.get_or_create_config() (line 356)
def get_or_create_config(self):
    """获取或创建归集配置"""
    # 1. 先尝试加载现有配置
    if self.config_exists():  # 检查 data/primary_targets.json 是否存在
        config, msg = self.load_config()
        if config:
            return config, msg, False  # 返回已有配置，is_new=False
    
    # 2. 配置文件不存在，生成默认配置
    config, msg = self.generate_default_config()
    return config, msg, True  # 返回新配置，is_new=True
```

### 2.2 默认配置生成逻辑

**`generate_default_config()` 方法 (line 198)：**

1. **数据源读取**：从 `output/analysis_cache/derived_data.json` 读取：
   - `family_units_v2`（优先）：包含完整的家庭单元信息（anchor, members, address）
   - `family_summary`（回退）：包含基本的家庭成员列表

2. **区分人员与公司**：
   ```python
   for name in profiles.keys():
       if self._is_company(name):  # 包含"公司"、"有限"、"集团"等关键词
           companies.append(name)
       else:
           persons.append(name)
   ```

3. **创建分析单元**：
   - 每个 `family_units_v2` 中的单元创建一个 `AnalysisUnit`
   - 主归集人（anchor）默认为户主
   - 成员关系从 `member_details` 获取（本人、配偶、子女等）

4. **未分配人员处理**：
   ```python
   # 关键修复：检查未分配的人员，为每个未分配人员创建独立分析单元
   unassigned_persons = [p for p in persons if p not in assigned_persons]
   for person_name in unassigned_persons:
       independent_unit = AnalysisUnit(
           anchor=person_name,
           members=[person_name],
           unit_type="independent",
           member_details=[...],
           note="系统自动为未分配人员创建的独立单元",
       )
       config.analysis_units.append(independent_unit)
   ```

5. **保存位置**：默认配置**不会立即保存到文件**，只是内存中返回给前端显示。

### 3.3 配置对报告的影响

用户调整的配置项会直接影响报告：

| 配置项 | 影响范围 |
|-------|---------|
| 家庭分组 | 报告中的家庭章节数量和组织方式 |
| 主归集人 | 每个家庭的 anchor（户主） |
| 成员关系 | 个人章节中的关系标注（本人、配偶、子女等） |
| 包含公司 | PART C: 公司核查部分的章节 |

## 四、问题3：这份报告是在何时生成的？

### 4.1 报告生成时机

| 场景 | 是否生成 | 触发方式 |
|-----|---------|---------|
| 分析完成时 (`run_analysis_refactored`) | ❌ 不生成 | - |
| 系统启动时 | ❌ 不生成 | - |
| **用户点击"生成报告"按钮** | ✅ **生成** | 前端调用 `/generate-html` |
| 定时任务 | ❌ 不生成 | - |

### 4.2 具体生成流程

```
用户点击"生成报告"
    │
    ▼
前端 ReportBuilder.tsx: generateReport()
    │
    ▼
POST /api/investigation-report/generate-html
    │
    ▼
后端实时生成：
  1. 加载 primary_targets.json（用户保存的配置）
  2. build_report_v5() → 生成JSON数据
  3. render_html_report_v3() → Jinja2模板渲染HTML
    │
    ▼
返回HTML给前端预览
    │
    ▼
用户点击"下载"或"保存" → 保存到 output/analysis_results/初查报告.html
```

### 4.3 生成耗时

- 报告是**实时生成**的，不是预先生成的
- 每次点击"生成报告"都会重新根据最新配置和数据生成
- 生成时间约 **5-30 秒**（取决于数据量）

## 五、问题4：程序默认会生成几份HTML报告？

### 5.1 答案：默认只生成 1 份 HTML 报告

### 5.2 分析完成时自动生成的文件

`run_analysis_refactored` (line 1400-1480) 自动生成的文件：

## 六、关键API端点汇总

| 端点 | 方法 | 功能 |
|-----|------|------|
| `/api/primary-targets` | GET | 获取当前归集配置（不存在则生成默认） |
| `/api/primary-targets` | POST | 保存归集配置 |
| `/api/primary-targets/entities` | GET | 获取可用实体列表（人员和公司） |
| `/api/primary-targets/generate-default` | POST | 重新生成默认配置 |
| `/api/investigation-report/generate-html` | POST | 生成HTML报告（所见即所得） |
| `/api/investigation-report/save-html` | POST | 保存HTML报告到服务器 |

## 八、总结

1. **默认归集**：从 `analysis_cache/derived_data.json` 自动识别家庭关系生成
2. **配置保存**：前端修改后保存到 `data/primary_targets.json`
3. **报告生成**：用户点击"生成报告"时实时生成，使用保存的配置
4. **HTML份数**：默认不自动生成，用户主动生成时只有 **1份**
