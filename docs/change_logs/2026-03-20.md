# 2026-03-20 工作日志

## 主题

围绕“原始数据 → 数据清洗 → 缓存落地 → 数据穿透/画像/正式报告”链路做一轮复核，重点修复本人互转、账户类别误判、工资识别对齐和缓存语义产物刷新问题。

## 今日完成

### 1. 数据清洗阶段复核与边界修正

- `data_cleaner.py` 修复“整批日期缺失”场景：
  - 当文件所有记录均无有效日期时，直接返回空结果
  - 明确写出 `invalid_rows`、`valid_rows`、`removed_rows`
  - 避免无效记录继续混入后续清洗合并
- `clean_and_merge_files()` 改为跳过空清洗结果参与 `concat`，避免 FutureWarning 和统计口径漂移
- 账户类别识别改为仅基于“本方账户属性”判断，不再因为对手方名称含“公司/企业”就把个人卡误判为对公账户
- 对短位纯数字账号仍保留对公结算账户识别，兼顾真实对公账号样本

### 2. 本人互转与工资识别继续收紧

- `financial_profiler.py` 中 `calculate_income_structure()` 改为复用统一本人互转识别逻辑，而不是只用“对手方同名”判断
- 这样即使出现“空对手方 + 转存开户/销户转存”等摘要，也会被识别为本人互转，不再误计入外部收入
- 工资识别结果继续向收入分类传导，避免“同一发薪单位、不同摘要编码”的工资入账被落为未知收入
- `personal_fund_feature_analyzer.py` 同步使用更稳定的真实工资口径，保持个人资金特征与画像输出一致

### 3. 账户层画像与正式报告解释增强

- `financial_profiler.py` 新增 `account_layer_summary`，按以下层级拆分原始流水规模：
  - 个人账户层
  - 对公账户层
  - 联名账户层
  - 其他账户层
- 当同一主体同时存在个人账户和对公账户流水时，画像会明确提示：
  - 各层交易笔数
  - 各层原始流入/流出金额
  - 当前真实收入/支出仍为全账户汇总口径，审阅时需分层理解
- `investigation_report_builder.py`、`report_view_builder.py` 和模板继续补强个人/家庭/公司的财务缺口解释与附录正式章节内容，使报告叙述能直接反映上述分层信息

### 4. 缓存与报告清单刷新逻辑继续收口

- `api_server.py` 的报告清单读取逻辑新增“语义产物是否过期”的判定：
  - 若 QA 版本过期、解释字段缺失或报告逻辑更新较新，会自动刷新 `report_package`
- 重新整理“先刷新个人画像、再计算家庭汇总”的执行顺序，避免家庭总收支仍沿用旧缓存数值
- `time_anomaly_detector.py` 增强对中文列清洗结果的兼容，允许直接消费 `data_cleaner` 输出结构参与复核检测

### 5. 前后端语义消费继续对齐

- `dashboard/src/components/NetworkGraph.tsx` 继续接入：
  - financial gap explanation
  - semantic issue lookup
  - dossier highlights
  - 静态正式报告预览
- `dashboard/src/components/KPICards.tsx`、`Header.tsx`、`Sidebar.tsx`、`LogConsole.tsx`、`TabContent.tsx` 与 `AppContext.tsx` 继续收口报告中心、图谱、日志与语义导航的交互状态
