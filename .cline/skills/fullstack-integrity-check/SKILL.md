# 全栈一致性与 UX 逻辑审计专家 (Static Focus)

## 触发场景 (When to use)
- 当用户要求“检查界面逻辑”、“查找显示问题”或“代码审查”时。
- 当用户强调 **“不要跑测试”** 或 **“通过代码理解”** 时。
- 当需要检查 Loading 状态、错误处理、空数据兜底等 UX 细节时。

## 🛡️ 核心铁律 (CRITICAL RULES)
1.  **UI 视觉锁定**：严禁修改 CSS 样式、配色和布局结构。
2.  **优先静态分析**：默认情况下，**不要** 运行 Python 测试脚本，除非用户明确要求“进行动态测试”。
3.  **代码即真相**：通过阅读 `tsx` 代码逻辑来判断问题，而不是通过猜测。

---

## 审计工作流 (Audit Workflow)

### 阶段一：数据契约与截断风险 (Data & Overflow)
**目标**：检查后端返回的数据，前端是否完整展示，有无“显示不全”的风险。
1.  **扫描后端**：阅读 `api_server.py`，列出 API 返回的所有字段（如 `full_name`, `amount`, `remark`）。
2.  **扫描前端**：阅读 React 组件（如 `KPICards.tsx`, `Sidebar.tsx`）。
3.  **检查点**：
    - **字段遗漏**：后端返回了 `description`，但前端组件里从未引用该字段？
    - **格式化缺失**：金额字段（`amount`）是否使用了 `toLocaleString()` 或 `formatCurrency`？如果没有，标记为“不符合惯用场景”。
    - **长文本风险**：对于可能很长的字段（如公司名、备注），前端是否处理了换行或截断（`truncate`, `overflow-hidden`）？如果没有，标记为“可能破坏布局”。

### 阶段二：UX 交互逻辑审计 (UX Logic Check)
**目标**：检查是否符合标准的 React 应用交互规范。
1.  **Loading 状态**：
    - 检查 API 调用处。在数据返回之前，界面是否有 `<Skeleton />` 或 `isLoading` 状态？
    - *风险*：如果没有，用户会看到一瞬间的空白或 `0`，体验极差。
2.  **Empty 状态 (空数据兜底)**：
    - 检查 `map()` 渲染列表的地方。
    - *风险*：如果数组为空（`data.length === 0`），界面是一片空白，还是显示了“暂无数据”？
3.  **Error 状态 (错误边界)**：
    - 如果 API 报 500，前端是白屏崩溃，还是弹出了 `Toast` 或显示错误提示？

### 阶段三：API 一致性 (原有功能)
1.  快速比对 `api.ts` 和 `api_server.py` 的 URL 和 Method 是否一致。
2.  **自动修复**：仅修复明显的拼写错误或字段映射错误。

### 阶段四：实战模拟 (仅在被要求时执行)
- **只有当用户指令中包含“运行测试”或“动态验证”时**，才生成并运行 `debug_integration_test.py`。
- 否则，跳过此步骤，节省内存。

## 输出报告格式 (中文)
请按以下结构输出代码审计报告：

### 1. 🚨 发现的 UX/逻辑 隐患
- **[组件名]**：具体问题描述。
  - *例如*：`Sidebar.tsx` 中 `user_list` 渲染时没有处理空数组，建议增加 `<EmptyState />`。
  - *例如*：`KPICards.tsx` 中金额显示为纯数字 `1000000`，未格式化为 `¥1,000,000.00`，不符合金融场景。
  - *例如*：后端返回了 `risk_level` 字段，但前端完全没用到。

### 2. 🔧 自动修复记录
- 列出你已经默默修正的代码逻辑（如字段拼写修复）。

### 3. 💡 优化建议
- 针对代码逻辑的改进建议。
